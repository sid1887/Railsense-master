STAGE 5 — Halt detection & traffic analysis improvements

Goal: Make halt detection robust (multi-sample, compare to scheduled station) and traffic detection cluster-based.

Files to edit:

services/haltDetection.ts

services/trafficAnalyzer.ts

services/predictionEngine.ts

Halt detection algorithm:

Pull last N snapshots for a train (from trainHistory)

Conditions for unexpected halt:

average speed in last M samples <= 1 km/h

distance between current coordinate and scheduled nearest station > stationProximityThreshold (e.g., >200m) OR current time != scheduled stop window

stationaryDuration > minStationarySeconds (e.g., 120s)

Return { halted: boolean, haltDurationSec, sectionId, isScheduledStop }

Traffic analyzer algorithm:

For a given coordinate, query trainHistory for all trains in last X minutes (e.g., 10 min)

Compute haversine distances and count trains within radius r (e.g., 5 km)

Return congestionLevel: LOW / MEDIUM / HIGH and list of nearby trains

Copilot prompt (Stage 5):

Implement services/haltDetection.ts:
- export async function analyzeHalt(trainNumber) that:
  - queries last 10 snapshots for train via trainHistory
  - computes if halted using a sliding-window stationary rule
  - uses station coordinates from static schedule (data.gov.in) to detect scheduled stops (create helper to load routes from JSON if static)
  - returns a result object { halted, haltDurationSec, nearestStation, isScheduledStop, samples }
Implement services/trafficAnalyzer.ts:
- export async function analyzeTraffic(lat,lng,minutesWindow=10) that:
  - queries trainHistory.getRecentPoints(minutesWindow)
  - calculates distances to each point via haversine function
  - returns nearbyTrains array and congestionLevel per heuristics:
      0-1 -> LOW, 2-4 -> MEDIUM, 5+ -> HIGH
Implement services/predictionEngine.ts:
- export function estimateWait(haltResult, trafficResult, weatherFlags)
- Use a deterministic formula:
  baseWait = historicalMedianWaitForSection || 10
  add = trafficFactor + weatherFactor + (haltDuration > baseWait ? 0 : 0)
- return { minWait, maxWait, method: 'heuristic' }

Test:

Use the snapshot DB entries from Stage 4 and run analyzeHalt() and analyzeTraffic() to see logical outputs.

FINAL NOTES & tips for best success with Copilot

One file at a time: Copilot produces better output when asked to create a single file/component with clear requirements and example inputs/outputs.

Provide sample JSON: For the scraper prompt, paste a small sample of the HTML or JSON you obtained from RailYatri into Copilot so it can craft accurate parsing code. If you don’t have HTML, Copilot can create a robust regex-based extractor; still expect manual tuning.

Run tests often: After generating each file, run it and fix small runtime errors — Copilot may create small TypeScript typos or import paths.

Lock UI style: Provide Copilot a Tailwind color palette and a sample component to copy style from; that keeps design consistent.

Fallback to mock: Ensure mockProvider always exists and is first-load tested so the app works with zero network access.

Respect scraping rules: throttle and cache; do not hammer public sites.
